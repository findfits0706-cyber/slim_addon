# Find Pilates 入会受付・SLIM転記連携 最終設計

## 1. 目指す状態

本案件の目的は、単なるフォーム改修やコピー補助ではない。

**申込者が入力した情報を一度だけ正しく受け取り、管理画面で確認・補正し、フロントスタッフが対象者を一度選択した後は、SLIMの現在画面へ一括転記して、例外だけを確認すれば登録できる状態**をつくる。

設計原則は次の4点とする。

1. **入力は一度、判断は一か所**  
   料金、コース、登録方式、SLIM操作順を複数箇所へ重複実装しない。
2. **通常時は一画面一操作**  
   初期版は、SLIM各画面で「この画面へ一括転記」を1回押し、SLIMの最終登録ボタンだけスタッフが押す。
3. **例外を隠さない**  
   既存値との差異、対象欄なし、コース候補不一致、ログイン切れなどは自動上書きせず、その場で明示する。
4. **後から完全自動化できる構造**  
   申込ごとの処理を「順序付き操作キュー」として保存し、画面遷移・登録ボタン操作を後から追加できるようにする。

## 2. 現状調査で判明した重要事項

### 2.1 WordPressへ移行しない

現行リポジトリには、PHP 8.3向けの独自アプリとして以下が既に存在する。

- PDOによるMySQL接続
- 管理者ログイン
- CSRF対策
- セッション管理
- 管理画面共通UI
- 監査ログの既存パターン

SLIM連携にWordPress固有機能は不要であり、移行すると認証・更新・プラグイン・API権限の管理対象が増える。したがって、**既存の独自PHP構成を拡張する**。

### 2.2 入会申込は現在MySQLではなくJSON保存

現行の入会受付は `admission/tmp/admissions.json` を読み書きしている。MySQLダンプには体験予約関連テーブルと管理者テーブルはあるが、入会申込テーブルはない。

入会フォームは未運用のため、今が最も安全にMySQLへ一本化できる時点である。Edge拡張用APIを作る前に、入会申込・写真・SLIM進捗をMySQLへ移す。

### 2.3 現行料金・選択肢に不整合がある

現行コードには以下が残っている。

- ダブル会員 13,200円
- 併用ベーシック 4,400円
- 併用ダブル 8,800円
- ウィークエンド会員
- 初月利用回数を申込者が手動選択する仕様
- 既存本館会員でも本館初月会費が初回計算へ入る可能性

確定仕様は以下とする。

- ベーシック：8,800円、月8回
- ダブル：12,650円、月16回
- 併用ベーシック：3,850円、月8回
- 併用ダブル：7,700円、月16回
- ウィークエンドは新規選択肢から完全削除
- Pilates部分は利用開始日の週区分から自動計算
- 本館へ同時入会する場合だけ、本館会費を既存の日割り計算
- 既存本館会員の本館会費は初回請求へ重複加算しない

### 2.4 SLIMはNuxt/Vue系の独自UI

保存HTMLから、SLIMは通常のHTML `select` だけではなく、readonly inputをクリックして候補を選ぶ独自コンポーネントを多用している。

そのため、単なる `element.value = ...` では不十分な可能性が高い。転記エンジンは次の順で処理する。

1. 対象欄を安定したIDまたはラベル基準で特定
2. 空欄であることを確認
3. 通常inputはネイティブsetter＋`input/change/blur`
4. 独自selectはクリックして完全一致の候補を選択
5. 独自日付欄は日付ピッカーを操作
6. 表示値を再読込して検証
7. 不一致なら失敗として停止・強調表示

自動生成された `data-v-*` 属性、DOMの何番目といった壊れやすいセレクタは使用しない。

### 2.5 新規入会は「入会受付」画面内で基本情報まで入力できる

保存HTMLでは、SLIMの「入会受付」ページに次の折りたたみ領域が含まれている。

- 入会項目
- 基本項目
- アンケート他
- その他画像

したがって、通常の新規入会は、別の「会員情報」画面を必須工程にせず、まず「入会受付」一画面で必要項目を転記する構成が最短である。「会員情報」は登録後の確認・修正用の補助画面として扱う。

### 2.6 顔写真欄は未特定

保存HTMLにある2つのファイル選択欄は、どちらも「公的身分証明書」の表裏用であり、会員顔写真用ではない。申込者の顔写真をここへ自動投入してはならない。

初期版では以下に限定する。

- 認証付きAPIから写真を取得
- `Downloads/FIND-SLIM/<申込ID>.jpg` に保存
- 実際の会員顔写真登録欄を取得するまで自動設定は無効

### 2.7 未解決事項は拡張機能の解析モードで取得する

現時点の主な未解決事項は以下である。

- 「登録会員No.」の採番・入力ルール
- 登録成功画面、登録完了時のDOM
- SLIMの代表的なエラーDOM
- 実際の会員顔写真登録欄
- 支払サイクル等、クリック後に動的生成される候補の実値

これらを人の推測で埋めず、Edge拡張へ「現在のSLIM画面を解析」機能を先に実装して取得する。

## 3. 最終システム構成

```text
公開入会フォーム
    │
    │ HTTPS / サーバー側再検証
    ▼
MySQL（申込・写真・料金スナップショット・SLIM操作キュー）
    │
    ├── 管理画面
    │     ├─ 原入力と転記値の確認・補正
    │     ├─ 実手続日の設定
    │     ├─ SLIM進捗・要確認・操作履歴
    │     └─ Edge拡張ペアリングコード発行
    │
    └── 拡張機能専用API
          │ 短期Bearer Token / 最小データ
          ▼
Microsoft Edge サイドパネル
    │
    ├─ 申込IDで対象者選択・固定
    ├─ 現在のSLIM画面を判定
    ├─ この画面へ一括転記
    ├─ 写真ダウンロード
    ├─ 差異・エラー表示
    └─ 進捗・ログをAPIへ返却
          │
          ▼
SLIM SNG
```

### 採用技術

- 公開・管理・API：既存PHP 8.3構成
- DB：MySQL 5.7、utf8mb4、PDO、ネイティブプリペアドステートメント
- 拡張：Microsoft Edge / Manifest V3 / Side Panel / Service Worker
- 初期配布：検証PCで「展開して読み込み」
- 安定後：Edgeポリシーまたは非公開配布を検討
- Native Messaging：初期版では採用しない

## 4. 正しい登録方式とSLIM操作キュー

登録工程を固定ステップではなく、申込ごとの `operations[]` として生成する。

### 4.1 Find Pilates単体

| 申込プラン | 操作1 |
|---|---|
| ベーシック | 入会受付：151 / FP |
| ダブル | 入会受付：135 / FP2 |

### 4.2 既存本館会員への追加

| 申込プラン | 操作1 |
|---|---|
| ベーシック併用 | 追加届：145 / P3 |
| ダブル併用 | 追加届：146 / P3W |

既存本館会員番号が未入力の場合は「準備未完了」とし、追加届の一括転記を開始できない。

### 4.3 本館とFind Pilatesへ同時入会

| 公開フォームの本館種別 | 操作1：本館入会受付 | 操作2：本館オプション | 最終操作：Pilates追加 |
|---|---:|---:|---:|
| ファインドマスター | 80 / MA | なし | 145 または 146 |
| デイフリー | 130 / DF | なし | 145 または 146 |
| ナイト＆ホリデー | 74 / GEH | なし | 145 または 146 |
| ナイト＆ホリデー34才以下 | 133 / A34 | なし | 145 または 146 |
| ジムフリー | 140 / FM | なし | 145 または 146 |
| ジム＆プール | 140 / FM | 141 / P1 の追加届 | 145 または 146 |
| ジム＆スタジオ | 140 / FM | 144 / S1 の追加届 | 145 または 146 |

重要事項：

- ジム＆プール、ジム＆スタジオは、Pilates追加とは別に本館オプション追加届が必要
- ウィークエンドは新規生成対象に含めない
- 旧データに `weekend` が存在した場合は「旧プラン・自動転記不可」として停止
- 追加届の申込日：実際の店頭手続日
- 追加届理由：9999 / その他
- 利用開始日：申込フォームの利用開始希望日
- 支払サイクル：月払い

### 4.4 コース選択の安全策

SLIM画面では数値IDではなくコード・名称が候補表示されるため、各コース設定に以下を持たせる。

```php
[
  'course_id' => 151,
  'code' => 'FP',
  'business_label' => 'ピラティスベーシック会員',
  'slim_option_texts' => [
    'FP：ファインドピラティス'
  ],
]
```

- 完全一致の許可リストだけで選択
- 部分一致、先頭一致、候補番号、曖昧検索は禁止
- 許可候補が0件または複数件なら停止

## 5. 公開入会フォームUI

既存の暖色系ベージュ・ブラウン、白いカード、十分な余白という方向は維持する。全面的な見た目の作り直しより、業務ルールと入力精度を優先する。

### STEP 1 利用方法・プラン

1. 「Find Pilatesのみ」「本館と併用」の大きな2カード
2. 併用時だけ「既存本館会員」「本館も同時入会」を表示
3. 本館プランは以下のセクション見出しで分類
   - FULL
   - 利用時間で選ぶ
   - 利用エリアで選ぶ
4. ウィークエンドは表示しない
5. 初月利用回数の選択欄は削除
6. 利用開始日により週割割合・回数を自動表示

### STEP 2 利用開始日・店頭手続き希望

- 利用開始希望日
- 第1希望：必須
- 第2・第3希望：設定で必須化可能
- 日付＋時間帯を1カードで表示
- 過去日、重複日時、休館日を防止
- 日曜日は夕方以降を表示しない
- 実手続日はここでは取得しない。管理画面で確定する

### STEP 3 お客様情報

SLIMと一致するよう、以下を変更する。

- 姓・名を分割
- セイ・メイを分割
- 生年月日：`input type=date` を第一選択、非対応時だけ年月日select
- 性別：必須、男性・女性・その他
- 電話番号種別：必須、携帯TEL・自宅TEL
- 電話番号
- メールアドレス
- 郵便番号、都道府県、住所1、住所2、住所3に対応できる分割
- 緊急連絡先氏名・続柄・電話
- 未成年時の保護者氏名

公開フォームではSLIMの最大長を超えないよう、姓・名各28文字、住所1～3各30文字相当のサーバー側検証を設ける。

### STEP 4 健康状態・入会資格

健康情報は申込・管理画面だけで扱い、Edge拡張APIへは返さない。

### STEP 5 規約・顔写真

- 写真の形式・容量・実体をサーバー側検証
- 保存先は公開ディレクトリ外
- 公開URLを直接持たせず、管理者認証または短期署名URLで配信

### STEP 6 確認・送信

- 申込IDを発行し、完了画面と自動返信へ表示
- 料金はサーバー側で再計算した値のみ表示・保存
- 二重送信防止用の一意キーを使用

### 料金計算

| 開始日 | 割合 | 8回プラン | 16回プラン |
|---|---:|---:|---:|
| 1～7日 | 100% | 8回 | 16回 |
| 8～14日 | 75% | 6回 | 12回 |
| 15～21日 | 50% | 4回 | 8回 |
| 22日～末日 | 25% | 2回 | 4回 |

1円未満は四捨五入（PHPでは `PHP_ROUND_HALF_UP` を明示）する。

例：

- ダブル12,650円 × 75% = 9,488円
- ダブル12,650円 × 25% = 3,163円
- 併用ベーシック3,850円 × 75% = 2,888円
- 併用ベーシック3,850円 × 25% = 963円

## 6. 管理画面UI

### 6.1 一覧

一覧は既存の左ペイン方式を維持し、以下を追加する。

- 申込対応ステータス
- SLIMステータス
- 操作進捗 `1/3` など
- フィルター：未着手、登録中、要確認、登録済み
- 処理中ロック表示

### 6.2 詳細上部

最初に次の情報だけをまとめる。

- 申込ID
- 申込日時
- 氏名
- 登録方式
- 実際の手続日
- 利用開始日
- SLIM会員番号
- 電話番号種別
- 不足・矛盾
- Edge連携状態

### 6.3 原入力と転記値

スタッフが修正した場合に区別できるよう、次の二層を持つ。

- 申込者入力：変更不可の原本
- SLIM転記値：スタッフが補正可能

差異がある項目だけ「申込者入力 → 転記値」を表示し、通常は転記値だけを見せる。

### 6.4 SLIM操作キュー

例：ジム＆プール＋ベーシック

```text
1. 本館入会受付        FM / 140        準備完了
2. 本館オプション追加  P1 / 141        1完了後
3. Pilates追加         P3 / 145        会員番号取得後
```

各操作に以下を保存する。

- 順序
- 対象SLIMページ
- コースID・コード・候補文字列
- 状態
- 転記日時・担当者
- 登録完了日時・担当者
- 最終エラー
- 再実行回数

### 6.5 Edgeペアリング

管理画面に「Edgeを接続」ボタンを置き、5分有効・1回限りのペアリングコードを発行する。拡張機能はコードを短期トークンへ交換する。

## 7. EdgeサイドパネルUI

推奨幅は400～440px。SLIM本体を隠さず、縦方向に流す。

### 7.1 未選択状態

```text
FIND × SLIM
接続中：スタッフ名

[今日予定] [未登録] [登録中] [検索]

申込ID                  申込日時
adm_2026...             6/25 14:32
adm_2026...             6/25 15:10
```

候補一覧はユーザー指定どおり、原則として申込IDと申込日時だけを表示する。

### 7.2 選択後

```text
対象会員を固定中
氏名：山田 花子
申込ID：adm_...
登録方式：本館同時＋ベーシック
実手続日：2026/06/25
利用開始日：2026/07/01

処理 1 / 3
本館入会受付  FM / 140

現在のSLIM画面：入会受付 ✓

不足・警告：なし

[ この画面へ一括転記 ]

転記結果
  入力 12
  一致 3
  既存値差異 0
  対象欄なし 1

次：内容確認後、SLIMの「登録」を押してください

[SLIM登録完了として記録]
```

### 7.3 選択固定

誤対象への転記を防ぐため、選択後は明示的に「対象を変更」するまで固定する。タブ移動・SLIM画面遷移でも保持する。ただし、ブラウザ終了・トークン失効・一定時間無操作で個人情報を消す。

### 7.4 一括転記時の規則

- 空欄だけ入力
- 既存値が正規化後に同じなら「一致」としてスキップ
- 既存値が異なる場合は上書きせず警告
- 見つからない欄は警告
- 必須欄が埋まらない場合は成功扱いにしない
- 画面が違う場合は実行ボタンを無効化
- 入力後に各欄を再読込して検証
- エラー欄をSLIM上で枠表示
- 結果をAPIへ送るが、入力値全文はログへ送らない

### 7.5 ログイン切れ

ログインページを検知したら次だけ表示する。

```text
SLIMのログインが切れています。
SLIMへ再ログインした後、このパネルで「再判定」を押してください。
```

SLIMのID・パスワードは保存・自動入力しない。

### 7.6 写真

- `Downloads/FIND-SLIM/<申込ID>.jpg`
- SLIM会員番号確定後は、必要に応じて再取得時に `<会員番号>.jpg`
- 氏名はファイル名に含めない
- 初期版では「顔写真をダウンロード」「フォルダで表示」まで
- 公的身分証明書欄へは設定しない

## 9. API設計

基本パス例：`/api/v1/extension/`

- `POST pair`：ペアリングコード交換
- `GET me`：接続スタッフ確認
- `GET admissions?scope=today|unregistered|in_progress|search&q=`
- `GET admissions/{applicationId}/transfer`
- `POST admissions/{applicationId}/lock`
- `POST admissions/{applicationId}/heartbeat`
- `DELETE admissions/{applicationId}/lock`
- `POST operations/{operationId}/fill-result`
- `POST operations/{operationId}/complete`
- `POST admissions/{applicationId}/member-number`
- `POST admissions/{applicationId}/photo-token`
- `GET photos/{signedToken}`

### 認証

- 管理画面ログイン済みスタッフが5分有効の一回限りコードを発行
- 拡張が8時間有効のBearer tokenへ交換
- DBにはトークンのハッシュだけを保存
- 失効可能
- 拡張ではセッション領域に保持し、ブラウザ終了時に消す
- 固定APIキーを拡張ソースへ埋め込まない

### APIレスポンスの最小化

返してよいもの：

- SLIM転記に必要な本人情報
- コースと操作キュー
- 日付
- 写真有無
- 進捗

返さないもの：

- 健康情報
- 管理メモ
- SLIMログイン情報
- DB情報
- 他申込者の不要な個人情報

全レスポンスに `Cache-Control: no-store` を付ける。

## 10. 実装順序

## 11. 公開前の停止条件

次の場合は自動転記せず、要確認にする。

- SLIM画面が不明
- 対象会員が固定されていない
- 別スタッフが処理中
- 実手続日がない
- 利用開始日がない
- 電話番号種別がない
- 既存会員なのに会員番号がない
- コース候補が完全一致しない
- 既存値が異なる
- 必須欄を検出できない
- 登録会員No.のルールが未確定で入力が必要
- ログイン切れ
- APIトークン失効

この「止まる設計」が、誤登録を防ぎながら操作数を減らすための中心となる。
